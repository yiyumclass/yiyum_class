# 어드민: 주문 · 결제 조회 설정

> 현재 서비스는 결제 연동 전 무료 신청 모드다. 이 화면은 무료 신청으로 발급된
> 이용권을 주문 원장처럼 조회하고, 실제 승인·취소·환불은 결제사 연동 후 확장한다.

## 구현 범위

- `/admin/orders` 신청·주문 목록
- 전체 신청, 오늘 신청, 이용 가능 건수와 확인된 결제액 요약
- 회원·상품·신청 번호 검색
- 무료 신청·결제·관리자 지급 경로 필터
- 전체·오늘·최근 7일·최근 30일 기간 필터
- 이용 가능·회수 상태 필터
- 관리자만 회원 이메일과 이용권 내역을 조회하는 보안 함수

## 데이터베이스 설정

아래 마이그레이션을 Supabase SQL Editor에서 실행한다.

`supabase/migrations/20260715160000_create_admin_order_ledger.sql`

이 파일은 `20260715150000_create_free_enrollments.sql` 실행 후 적용해야 한다.
생성되는 `get_admin_order_ledger()` 함수는 다음 원칙을 따른다.

- 로그인한 활성 관리자만 결과를 조회한다.
- 일반 회원은 다른 회원의 이메일이나 신청 내역을 조회할 수 없다.
- 무료 신청과 관리자 지급 금액은 0원으로 표시한다.
- 실제 결제 출처의 금액은 주문 테이블 도입 전까지 임의로 계산하지 않고 `연동 대기`로 표시한다.
- 콘텐츠 접근 여부는 주문 상태가 아니라 최종 이용권 상태를 기준으로 표시한다.

## 화면 확인

관리자 계정으로 `/admin/orders`에 접속한다.

- 사이드 메뉴의 `주문 · 결제`가 활성 링크여야 한다.
- 마이그레이션 적용 후 상단에 `운영 데이터`가 표시되어야 한다.
- 무료 콘텐츠를 신청한 회원의 이름, 이메일, 상품, 신청 시각과 이용권 상태가 표시되어야 한다.
- 검색과 경로·기간·상태 필터를 함께 적용할 수 있어야 한다.
- 신청 내역이 없으면 오류 대신 빈 상태 안내가 표시되어야 한다.

## 원장의 정식화

- `20260716120000_create_orders.sql` 적용 후, 주문 원장은 별도 `public.orders` 테이블에서
  조회한다. 무료 신청(`claim_free_product`)과 관리자 지급(`admin_grant_product_entitlement`)은
  이용권을 upsert하면서 주문 행을 **append-only**로 쌓는다. 이용권은 여전히 회원·상품당 한 행이지만,
  주문은 신청·지급마다 새 행으로 쌓이므로 같은 회원이 같은 상품을 여러 번 신청하면 여러 주문 행이 나온다.
- 주문 행은 시점 확정 금액(`amount`)을 보존한다. 무료 신청은 0원, 관리자 지급은 상품의 표시가로 기록되며,
  화면의 `결제 금액`은 더 이상 `연동 대기`가 아니라 실제 확정 금액을 표시한다.
- 최초 발급 시각은 이용권의 `granted_at`(재신청 시 보존)과 각 주문의 `created_at` 양쪽에 남는다.
  기존 이용권은 마이그레이션의 backfill로 주문 행이 1회 생성된다(이용권 상태 `active`→주문 `paid`,
  `revoked`→`canceled`).
- 남은 범위는 **유료 결제의 상태 전이(`pending`→`paid`)와 취소·환불(`canceled`/`refunded`) 연동**이다.
  `orders`에는 `status`, `payment_key`, `approved_at`, `canceled_at` 컬럼이 준비되어 있으나,
  실제 승인·취소·환불을 기록하는 라우트와 웹훅은 결제사 연동 시 붙인다.

## 결제 연동 후 확장 범위

1. ✅ 주문 시점 금액을 보존하는 `orders` 테이블 (`20260716120000_create_orders.sql`)
2. 결제 승인 번호와 결제 수단 (`orders.payment_key` 채우기)
3. 승인 실패·취소·환불 상태와 처리 시각 (`status`/`approved_at`/`canceled_at` 전이 로직)
4. 결제사 웹훅 기반 상태 동기화
5. 취소·환불 작업의 관리자 권한 검사와 감사 로그
